# Rudder Meta

This skill is for local Rudder users who have already run `rudder init` in a project. After reading it, an AI should understand the Rudder architecture, operating model, and customization entry points inside that user project, then modify the generated `.rudder/` and platform directory files according to the user's request.

The default operating scope is local files in the user project:

- `.rudder/`: workflow, config, tasks, spec, workspace, scripts, and runtime state.
- Platform directories: `.claude/`, `.codex/`, `.cursor/`, `.opencode/`, `.kiro/`, `.gemini/`, `.qoder/`, `.codebuddy/`, `.github/`, `.factory/`, `.pi/`, `.kilocode/`, `.agent/`, `.windsurf/`, and similar directories.
- Shared skill layer: `.agents/skills/`.

Do not assume the user has the Rudder source repository. Do not default to modifying the global npm install directory or `node_modules`.

## How To Use

1. Read `references/local-architecture/overview.md` first to establish the local Rudder system model.
2. If the request involves a specific AI tool, read `references/platform-files/platform-map.md` and the relevant platform file notes.
3. If the user wants to change behavior, read `references/customize-local/overview.md`, then open the specific customization topic.
4. Before editing, read the actual files in the user project and treat local content as authoritative.

## Current Rules

- `.rudder/workflow.md` is the local workflow source of truth.
- `.rudder/config.yaml` is the project-level Rudder configuration and task hook configuration entry point.
- `.rudder/spec/` stores the user's project-specific coding conventions and design constraints.
- `.rudder/tasks/` stores task PRDs, technical notes, research files, and JSONL context.
- `.rudder/workspace/` stores developer journals and cross-session memory.
- Platform settings/config files decide which hooks, agents, skills, commands, prompts, and workflows actually run.
- `.rudder/.template-hashes.json` and `.rudder/.runtime/` are management/runtime state files. Confirm necessity before editing them.

## Do Not

- Do not treat Rudder upstream source code as the default target for local customization.
- Do not modify the global npm install directory or `node_modules/@mengde1231/rudder` to implement project needs.
- Do not overwrite user-modified local files with default templates.
- Do not put team-private project rules into the public `rudder-meta`; put project rules in `.rudder/spec/` or a project-local skill.
- Do not describe removed historical mechanisms as current Rudder behavior.
